# M30.1 — Backup Schedules + Remote Destinations

**Status:** drafted
**Builds on:** M30 (ADR-0075). Local-only restic repo + manual backup + retention timer already shipped.
**ADR target:** **0077** — backup destinations + scheduling model.

## Goal

Add two admin-managed capabilities on top of M30:

1. **Multiple restic destinations** (local always; admin-defined remotes: SFTP / S3-compat / B2 / Azure / GCS / REST server). No rclone — restic-native only.
2. **Scheduled backups** — admin-defined cron schedules per kind (account_backup / system_backup) and per user, each linked to one or more destinations. After local backup completes, `restic copy` mirrors the snapshot to every linked remote **asynchronously** (does not block the backup job).

Out of scope:
- Per-user destination management (admin only).
- User-side scheduling UI (admin only).
- Auto-created defaults on user-add.
- Per-destination retention policies (single policy across all repos in v1; revisit if real demand surfaces).
- rclone / Google Drive / Dropbox / OneDrive (operators wanting these run their own restic-rest-server or S3 gateway).

## Constraints + invariants

- **Restic-native only.** Backend menu = `local | sftp | s3 | b2 | azure | gcs | rest`. S3 covers AWS, Backblaze B2 (S3 API), Wasabi, MinIO, Cloudflare R2.
- **Credentials never in DB.** `backup_destinations.credentials_ref` is a path under `/etc/jabali-panel/restic-remotes/<dest-id>.env` (root:root 0600). Each env file holds the restic env vars for that backend (`AWS_ACCESS_KEY_ID`, `B2_ACCOUNT_ID`, etc.) plus the dest's repo password if rotated independently. DB only stores the file pointer.
- **Local repo is implicit.** Schedule with zero linked destinations = local-only. `restic copy` to local target = no-op (skip).
- **Async copy.** Local backup marks job `succeeded` first. A separate `backup_copy_jobs` queue row per (backup_job, destination) is enqueued and processed by the copy worker. Backup UI shows local status + per-destination copy status.
- **Single repo password across all remotes.** `restic copy --from-repo … --from-password-file=…` requires source + dest passwords. v1 reuses `/etc/jabali-panel/restic-repo.password` for all destinations (same password, different storage). Per-destination password rotation = M30.2.
- **Cron parser:** `github.com/robfig/cron/v3` with `cron.WithSeconds()` disabled. Standard 5-field cron. Presets render to canonical strings: Daily=`0 3 * * *`, Weekly=`0 3 * * 0`, Monthly=`0 3 1 * *`.
- **Tick cadence:** every 60s. Single systemd timer `jabali-backup-scheduler.timer` → `jabali backup scheduler tick`. Idempotent; missed ticks (host suspend) are tolerated — the next tick picks up overdue schedules.
- **Concurrency:** scheduler dispatches via panel-agent the same way manual backups do. Existing per-(kind,user-id) job lock in panel-agent rejects duplicates (409); scheduler logs and skips on overlap.
- **Copy worker = systemd-run transient unit per copy job** (M29 carry-over). Survives `jabali update`. Unit name `jabali-backup-copy-<copy-job-id>.service`. ProtectHome=read-only; ReadWritePaths covers the local repo + cache.
- **Snapshot tagging carries through copy.** `restic copy` preserves tags. Retention timer scans by `--tag jabali` on every dest separately.
- **Retention runs per-destination.** v1 timer runs `restic forget --keep-* --prune` against the local repo. v2 add: enumerate destinations + run retention against each. Same `--keep-*` knobs from `server_settings`.

## ADR-0078 decision points

1. Restic-native backends only (no rclone).
2. Async copy via `restic copy` after local success.
3. Admin-only destination + schedule management.
4. Single shared repo password across destinations in v1.
5. Cron-string field (not separate hour/min/dow cols) — preset UI maps to canonical strings.
6. Scheduler = single 60s tick worker reading DB, not per-schedule systemd timers.

### `jabali backup scheduler tick`

```go
func (s *Scheduler) Tick(ctx context.Context) error {
  due, err := s.repo.ListDue(ctx, time.Now())
  for _, sched := range due {
    if !sched.Enabled { continue }
    if running, _ := s.agent.IsBackupRunning(ctx, sched.Kind, sched.UserID); running {
      log.Infof("schedule %s: skip — backup already running", sched.ID)
      s.repo.UpdateNextRun(ctx, sched.ID, nextFireFromCron(sched.CronExpr, time.Now()))
      continue
    }
    if err := s.agent.DispatchBackup(ctx, BackupRequest{
      Kind: sched.Kind, UserID: sched.UserID, ScheduleID: sched.ID,
    }); err != nil {
      log.Errorf("schedule %s: dispatch failed: %v", sched.ID, err)
      continue
    }
    s.repo.MarkRan(ctx, sched.ID, time.Now(), nextFireFromCron(sched.CronExpr, time.Now()))
  }
  return nil
}
```

### Copy worker

After backup_jobs marks `succeeded`, panel-api hook (or post-backup callback) fans out:

```go
for _, dest := range schedule.Destinations {
  if dest.Kind == "local" { continue }   // no-op
  copyJob := backup.CopyJob{ID: ulid.Make(), BackupJobID: bj.ID, DestinationID: dest.ID, Status: "queued"}
  copyJobs.Create(copyJob)
}
```

Worker tick (60s timer, separate from scheduler):

```go
func (w *CopyWorker) Tick(ctx context.Context) error {
  queued, _ := w.repo.ListQueued(ctx, 5)   // 5 concurrent max
  for _, cj := range queued {
    unitName := fmt.Sprintf("jabali-backup-copy-%s.service", cj.ID)
    cmd := exec.Command("systemd-run",
      "--unit="+unitName, "--collect", "--quiet",
      "/usr/local/bin/jabali", "backup", "copy", "run", "--copy-job-id", cj.ID,
    )
    if err := cmd.Run(); err != nil { ... }
    w.repo.MarkRunning(ctx, cj.ID)
  }
  return nil
}
```

`jabali backup copy run --copy-job-id=<id>`:
```
restic --repo /var/lib/jabali-backups/repo \
       --password-file /etc/jabali-panel/restic-repo.password \
       copy \
       --to-repo <dest.URL> \
       --to-password-file /etc/jabali-panel/restic-repo.password \
       --tag job-id=<bj.JobID>           # only this job's snapshots
```
Env loaded from `dest.CredentialsRef` (`/etc/jabali-panel/restic-remotes/<dest-id>.env`).

## Risk + rollout

- **Credential leak surface.** Env files at 0600 root:root. UI never returns secrets in GET responses; PATCH with empty secret = no change. Documented in runbook.
- **Tick storm if 1000 schedules due simultaneously after host suspend.** ListDue caps at 50 per tick; remainder picked up next tick.
- **Copy failures cascade.** Failed copy jobs retry up to 3× via worker (exponential backoff: 1m, 5m, 30m); after 3 fails, status=failed and M14 notification fires.
- **`restic copy` requires both passwords.** v1 uses same password file for source + dest. If a remote rotates its password, current pattern breaks. M30.2 = per-destination password file at `/etc/jabali-panel/restic-remotes/<id>.password` mirroring the env file pattern.
- **Backwards compat.** All M30 manual flows preserved; schedules + destinations are pure additions. Existing manual backups create no copy jobs (no schedule = no destinations linked).

## Out of scope (future M30.2 / M30.3)

- Per-destination retention policies.
- Per-destination passwords / key rotation.
- Encryption-at-rest different from repo password.
- Bandwidth limits on copy (`restic` doesn't expose; would need traffic-shaping wrapper).
- Cross-region replication topology UI.
- rclone backends (Google Drive et al.) — operators run own restic-rest-server.
